# Tone Calibration

## Behavioral Change

Claude 4.6 (Opus and Sonnet) is significantly more responsive to system prompts than previous models. Urgency language that was necessary to prevent undertriggering in older models now causes overtriggering. Prompts designed for Claude 3.x or early 4.x that used aggressive emphasis (CRITICAL, MUST, ALWAYS, NEVER) now over-steer the model, producing rigid or excessive behavior.

## Impact on Generated Teams

Every agent prompt, skill prompt, and rule generated by A-Team is a system-level instruction. If these prompts carry over the urgency conventions from older models, generated teams will exhibit overtriggering -- agents using tools when unnecessary, following constraints too literally, or refusing to exercise judgment.

## Recommended Pattern

Use normal, direct language. Provide context for why a behavior matters instead of shouting.

| Old Pattern (Over-triggers on 4.6) | New Pattern |
|-------------------------------------|------------|
| `CRITICAL: You MUST use this tool when...` | `Use this tool when...` |
| `ALWAYS check X before proceeding` | `Check X before proceeding` |
| `NEVER skip this step` | `Do not skip this step because {reason}` |
| `IMPORTANT: You MUST ALWAYS...` | `{Action} because {reason}` |
| `WARNING: Failure to X will result in...` | `X prevents {consequence}` |

When emphasis is genuinely needed, use context-based motivation instead of capitalization:

```text
# Before (over-triggers)
CRITICAL: You MUST validate all inputs before processing. ALWAYS check for null values.

# After (appropriate triggering)
Validate all inputs before processing. Null values cause downstream failures in the pipeline, so check for them at system boundaries.
```

The explanation ("Null values cause downstream failures") is more effective than "CRITICAL" because Claude 4.6 generalizes from the reasoning.

## Anti-Patterns to Avoid

- Capitalizing entire words for emphasis (MUST, ALWAYS, NEVER, CRITICAL) -- use normal case with reasoning
- Stacking multiple urgency markers in one instruction -- one clear statement suffices
- Copying prompt patterns from Claude 3.x documentation without recalibrating tone
- Adding "IMPORTANT:" prefixes to every instruction -- dilutes genuine priority signals
